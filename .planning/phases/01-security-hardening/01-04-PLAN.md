  <action>
    Edit `server/src/lib.rs` in the `submit_answer` reducer.

    **Step 1: Add named constants** near the top of the file (after use statements, before struct definitions):
    ```rust
    const MAX_ANSWERS_PER_SESSION: usize = 80;
    const MIN_RESPONSE_MS: u32 = 200;
    const MAX_RESPONSE_MS: u32 = 120_000;
    ```

    **Step 2: Rename `_player` to `player`** in submit_answer (the `get_player(ctx)?` call near the top). The underscore prefix suppresses the "unused variable" warning — removing it is correct here because we will now use the player variable for the tier check.

    **Step 3: Add guards BEFORE the `ctx.db.answers().insert()` call.** Place them immediately after the session ownership and is_complete checks (which must be kept). The exact insertion point is: after `if session.is_complete { return Err(...) }` and BEFORE `ctx.db.answers().insert(...)`.

    ```rust
    // SEC-05: response_ms bounds
    if response_ms < MIN_RESPONSE_MS {
        return Err("Response time below minimum threshold".into());
    }
    if response_ms > MAX_RESPONSE_MS {
        return Err("Response time above maximum threshold".into());
    }

    // SEC-04: per-session answer cap
    let existing_count = ctx.db.answers()
        .iter()
        .filter(|ans| ans.session_id == session_id)
        .count();
    if existing_count >= MAX_ANSWERS_PER_SESSION {
        return Err("Session answer limit reached".into());
    }

    // SEC-06: (a, b) pair must be within player's learning tier
    let pair_tier = pair_learning_tier(a, b)
        .ok_or_else(|| "Invalid problem pair".to_string())?;
    if pair_tier > player.learning_tier {
        return Err("Problem pair above player's current tier".into());
    }
    ```

    CRITICAL: Do not place these guards after the Answer insert or after the ProblemStat update. The guards must be the first logic after the session validation block to prevent any partial state on rejection. Reference PITFALLS.md Pitfall 2.
