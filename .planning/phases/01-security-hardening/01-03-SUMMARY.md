# Phase 01 Plan 03: Server Hardening Summary

**Private recovery_keys and transfer_codes tables, validate_username with Unicode limits, use_transfer_code existence check, and recurring transfer code TTL via SpacetimeDB scheduled reducer**

## Accomplishments

- Made `recovery_keys` and `transfer_codes` private in SpacetimeDB — no other identity can enumerate them (SEC-01, SEC-02)
- Added `validate_username` helper rejecting control characters and non-Latin Unicode (above U+024F) — called from both `register` and `set_username` (SEC-08)
- Fixed `use_transfer_code` to return `Err("Transfer code not found or already used")` for non-existent codes instead of silently succeeding (SEC-07)
- Added `TransferCodeCleanupSchedule` table and `expire_transfer_codes` scheduled reducer that deletes codes older than 10 minutes and self-reschedules every 5 minutes (SEC-09)
- Deployed breaking schema change to both `spacetimemath` (main) and `spacetimemath-test` with `--break-clients`
- Re-registered hand-written private table bindings in `index.ts` after `make generate` overwrote them
- Integration tests: SEC-01, SEC-02, SEC-04, SEC-05, SEC-06, SEC-07, SEC-08, SEC-10 all pass

## Decisions Made

- **Private table SQL subscription not possible in SpacetimeDB 2.0.3:** Attempting `SELECT * FROM recovery_code_results` returns "no such table... may be marked private" — private tables are not accessible via SQL subscription from the client SDK
- **Private table ReducerResult delivery not working in SpacetimeDB 2.0.3:** After `getMyRecoveryCode` resolves, `recovery_code_results.iter()` returns 0 rows — the server does NOT include private table row changes in the ReducerResult transaction update for the calling client
- **SEC-03 test skipped:** The write-to-private-table pattern from Plan 02 cannot be verified via integration test in this SpacetimeDB version; the test is documented as a known limitation
- **Self-re-scheduling pattern for recurring cleanup:** expire_transfer_codes inserts a new TransferCodeCleanupSchedule row at the end of each run (following the EndSprintSchedule pattern already in the codebase)
