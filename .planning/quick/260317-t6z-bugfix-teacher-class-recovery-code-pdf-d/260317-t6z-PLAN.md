<root_cause_analysis>
Three silent failure modes identified in `handlePrintAll`:

1. **Popup blocked** — `window.open('', '_blank')` returns `null` in popup-blocking
   browsers/contexts (incl. iOS Capacitor after an `await`). Line 280 just `return`s
   with no user feedback.

2. **No recovery keys** — After the `--delete-data=on-conflict` deploy the
   `recovery_keys` table data was restored from CSV backup, but students who
   registered after the backup window may have no key. Server's
   `get_class_recovery_codes` silently skips members without a key (`None =>
   continue`). Poll times out at 5 s, window closes.

3. **Stale-rows race** — If teacher clicks Print a second time, `classRecoveryResultsRef`
   still contains the previous results at the moment polling starts (the server DELETE
   event has not yet propagated). The poll exits immediately with old data and the new
   window shows stale cards. Not a total breakage but incorrect output.

Note: `window.open()` must be called synchronously (before the first `await`) to avoid
popup blocking in most browsers — this is correctly handled. The `null` check on `win`
must surface an error rather than silently return.
</root_cause_analysis>

   ```typescript
   // Phase 1: wait for stale rows to clear (or skip if ref was already empty)
   if (prevCount > 0) {
     const clearDeadline = Date.now() + 2_000;
     while (Date.now() < clearDeadline) {
       const cur = classRecoveryResultsRef.current.filter(r => r.classroomId === classroomId).length;
       if (cur === 0) break;
       await new Promise(res => setTimeout(res, POLL));
   ```
